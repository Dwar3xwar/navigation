Changing Data

You'll often need to pass data when changing views/States in your application. In the email application example, when a user selects a mail from their inbox, you need to pass the id of that mail to the 'details' State.

You can pass data when navigating to a Dialog or along a Transition. The navigate function accepts a second parameter that holds this data. Assuming the current State is the 'listing' State of the 'inbox' Dialog, here's how you'd pass a mail id of 10 to the 'details' State: 

Navigation.StateController.navigate('select', { id: 10 });

Refresh Navigation

There's another situation in the email application where you need to pass data, but that doesn't involve changing views. Because inboxes can grow large, there are 'next' and 'previous' buttons on the UI, allowing the user to page through their mails. Clicking the 'next' button increments the page number and refreshes the inbox display. This kind of navigation, where the data changes but the State doesn't, is called refresh navigation.

Refreshing doesn't require a Transition because the State doesn't change. To refresh, you pass the new data to the refresh function on the StateController. Assuming the current State is the 'listing' State of the 'inbox' Dialog, here's how to move to the second page: 

Navigation.StateController.refresh({ pageNumber: 2 });

Routes

Navigation treats route and query string parameters equally. You can change the routes of the 'inbox' States, setting the id and pageNumber as route parameters, without impacting on the way the data is passed: 

var inboxDialog = {
    key: 'inbox', initial: 'listing', states: [
        { key: 'listing', route: '{pageNumber?}',
		trackCrumbTrail: false, transitions: [
            { key: 'select', to: 'details' }
        ]},
        { key: 'details', route: 'mail/{id}' }
    ]
};
